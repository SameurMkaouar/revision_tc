Chapitre 1

Concepts & Architecture

L'industrie des logiciels a changé

L'industrie du logiciel est passée des architectures monolithiques (tout en un seul bloc) aux microservices (services indépendants), pour gagner en rapidité, flexibilité et scalabilité.

Avant — Monolithique Actuellement — Microservices
App unique, toutes fonctionnalités imbriquées (BDD, logique, UI) Petits services indépendants, chacun dédié à une fonction
Cycles longs — redéploiement complet à chaque modification Déploiement et mise à jour indépendants par service
Mise à l'échelle lente — dupliquer toute l'app Scalabilité ciblée — seuls les services à fort trafic

Motivation : Pourquoi les conteneurs ?

Imaginons un projet avec Node.js, MongoDB, Redis et Ansible. Chaque outil a ses propres dépendances et versions — ce qui crée une série de problèmes appelée la Matrice d'Enfer :

  • Compatibilité OS — certains services ne fonctionnent pas avec la même version de l'OS
  • Conflits de bibliothèques — service A veut lib v1, service B veut lib v2
  • Mise à niveau risquée — vérification de compatibilité à chaque upgrade
  • Différences Dev / Test / Prod — comportement non garanti selon l'environnement
⚠️ The Matrix from Hell
Interdépendances complexes, conflits de versions, configurations spécifiques à gérer → la conteneurisation est la solution.

Solution : Les Conteneurs

Chaque composant (Node.js, MongoDB…) s'exécute dans un conteneur séparé avec ses propres dépendances, tous sur le même OS. On ne configure le moteur de conteneurisation qu'une seule fois.

📦 Définition
Un conteneur = application + dépendances + bibliothèques + binaires + config, dans un seul package isolé. Tous les conteneurs partagent le même noyau de l'hôte.
  • Environnements complètement isolés (processus, interfaces réseau, points de montage)
  • Un hôte Ubuntu peut exécuter des conteneurs Fedora, CentOS… car le noyau Linux est commun
  • Légers : un seul noyau pour toutes les instances → réduction de la consommation de ressources
💡 Avantages clés
Flexibles · Légers · Interchangeables · Portables (dev → cloud → prod sans friction) · Évolutifs (scalabilité ciblée)

Conteneur vs Machine Virtuelle

Une VM émule un ordinateur complet avec son propre OS sur un hyperviseur (virtualisation lourde). Un conteneur partage le noyau de l'hôte et isole uniquement l'application (virtualisation légère).

Critère Machine Virtuelle Conteneur
Isolation Complète — noyau + OS distincts Noyau de l'hôte partagé
Ressources Lourde (mémoire + stockage par instance) Légère (partage le noyau)
Démarrage Lent — doit charger un OS complet Rapide — quelques secondes
Performance Réduite (couche hyperviseur) Proche de l'hôte
Portabilité Dépend de l'hyperviseur Très portable (Docker compatible partout)
Sécurité Forte isolation entre VMs Moins d'isolation — noyau partagé
Cas d'usage OS multiples, isolation stricte Microservices, déploiements cloud

Namespaces

Sans namespaces, tous les processus partagent les mêmes ressources : même hostname, PIDs globaux, même réseau, mêmes montages. Les namespaces sont une fonctionnalité du noyau Linux qui permettent de diviser ces ressources en sous-environnements isolés — chaque conteneur obtient sa propre "vision" indépendante du système.

  • User — UID/GID isolés ; chaque conteneur a son propre root local sans privilèges réels sur l'hôte
  • PID — chaque conteneur a ses propres identifiants de processus ; le PID 1 d'un conteneur est son processus principal
  • Network — interfaces réseau, adresses IP et routes indépendantes par conteneur
  • Mount — arborescence du système de fichiers isolée ; le conteneur ne voit pas les montages des autres
  • UTS (UNIX Time-Sharing) — hostname et domaine DNS distincts par conteneur
  • IPC (Interprocess Communication / Communications Inter-Processus) — mémoire partagée, sémaphores et files de messages isolés entre conteneurs

Cgroups (Control Groups)

Les cgroups sont une fonctionnalité du noyau Linux permettant de limiter, comptabiliser et isoler l'utilisation des ressources (CPU, mémoire, disque, réseau) d'un groupe de processus. Ils garantissent qu'un conteneur ne peut pas monopoliser les ressources de l'hôte.

  • Limitation — fixe un quota max de mémoire, CPU, disque ou réseau par groupe
  • Comptabilité — mesure la consommation réelle de chaque groupe (utile pour la facturation)
  • Priorisation — certains groupes obtiennent une part plus grande de CPU ou de bande passante I/O
  • Isolation — chaque groupe fonctionne indépendamment sans interférer avec les autres
  • Contrôle — possibilité de geler l'exécution d'un groupe ou de sauvegarder/restaurer son état
💡 Namespaces + Cgroups = base des conteneurs
Namespaces isolent l'environnement (chaque conteneur voit son propre monde). Cgroups contrôlent les ressources (quota CPU, mémoire). Ensemble, ils constituent la fondation technologique de LXC (Linux Containers) et Docker.

C'est quoi Docker ?

Lancé en 2013 par Solomon Hykes (dotCloud), Docker est une plate-forme ouverte permettant d'empaqueter une application et ses dépendances dans un conteneur isolé, exécutable sur n'importe quel serveur. Docker a simplifié LXC (Linux Containers) en proposant une interface conviviale et des outils d'automatisation.

  • Virtualisation légère — noyau de l'hôte partagé, pas de noyau propre par conteneur
  • Namespaces & cgroups — isolation de l'environnement + contrôle des ressources
  • Copy-on-Write (COW) — optimisation de l'espace disque lors de la création de conteneurs
  • Déploiement instantané — pas besoin de démarrer un OS complet, création en quelques secondes
  • Adapté aux microservices — portable, scalable, léger, idéal pour le cloud

Architecture Docker

Docker repose sur une architecture client-serveur : le Docker Client envoie des commandes au Docker Daemon qui les exécute. Ils peuvent être sur la même machine ou communiquer en réseau.

  • Docker Client — interface utilisateur : Docker CLI (Linux) ou Remote API (Mac/Windows) ; envoie les commandes au daemon
  • Docker Daemon (dockerd) — cœur de Docker ; reçoit et traite les requêtes (gestion des conteneurs, images, volumes, réseaux)
  • REST API — canal de communication standardisé entre le client et le daemon
  • Docker Engine = Client + REST API + Daemon — permet de créer, exécuter et partager des conteneurs
  • Docker Registry / Hub — bibliothèque d'images Docker (public = Docker Hub, privé possible) ; le daemon y récupère les images nécessaires
  • Image — template statique immuable | Conteneur — instance active en cours d'exécution d'une image

Container Runtime : containerd, runc & libcontainer

Docker ne gère pas directement l'exécution des conteneurs — il délègue à une chaîne de runtimes spécialisés :

Chaîne d'exécution
Docker Daemon → containerd (haut niveau) → runc (bas niveau, conforme OCI) → libcontainer (noyau Linux)
  • containerd — runtime haut niveau ; gère le cycle de vie des conteneurs (lancement, arrêt, suppression), les images et volumes. Intégré à Docker, mais aussi utilisé seul par Kubernetes
  • runc — runtime léger bas niveau ; exécute le conteneur dans un environnement isolé via namespaces et cgroups. Conforme à la spec OCI Runtime
  • libcontainer — bibliothèque bas niveau liant runc directement au noyau Linux (namespaces, cgroups, interfaces réseau, pare-feu) sans dépendre de LXC (Linux Containers)
  • OCI (Open Container Initiative) — normes ouvertes garantissant l'interopérabilité : Runtime Spec (comment exécuter) & Image Spec (format des images)

Image vs Conteneur vs Dockerfile

  • Image Docker — modèle statique immuable contenant l'OS de base, l'application et toutes ses dépendances. Créée via un Dockerfile ou téléchargée depuis un Registry. Sert de base pour instancier un ou plusieurs conteneurs
  • Dockerfile — fichier texte contenant les instructions de construction d'une image (copier des fichiers, installer des packages, définir des commandes…)
  • Conteneur Docker — instance active et exécutable d'une image. Chaque conteneur est isolé et sécurisé ; actions possibles : exécuter, démarrer, arrêter, déplacer, supprimer
🔄 Cycle de vie d'un conteneur
Création (depuis une image) → Exécution → Arrêt → Suppression

Docker, initialement conçu pour Linux, est devenu multiplateforme : Windows, MacOS, AWS, Azure.

Distributions Linux supportées : CentOS, Debian, Fedora, Oracle Linux, RHEL, Ubuntu, openSUSE, SUSE.

  2. Lancer l'installateur (activer WSL 2 et Hyper-V)

groups

# Exemple: docker run hello-world

Syntaxe 2 (management commands)

Instanciation & Exécution des Conteneurs

1. Instanciation des images

Générer un conteneur à partir d'une image Docker :

  • Allocation des ressources — mémoire, CPU
  • Isolation de l'environnement — environnement virtuel séparé
  • Initialisation des processus — services définis par l'image
  • Connexion au réseau — communication avec d'autres conteneurs

2. Exécution des conteneurs

  • Téléchargement de l'image — si absente, récupérée depuis Docker Hub
  • Création du conteneur — instance isolée prête à exécuter
  • Exécution de la commande — définie lors du lancement
💡 Bonne pratique
Un conteneur devrait exécuter un seul processus. Pas de SSH, pas de systemd, pas de processus init.
Chapitre 3

Gestion des Images

Qu'est-ce qu'une image Docker ?

Une image Docker est un modèle/gabarit prêt à l'emploi pour créer des conteneurs. Elle contient :

  • Le système d'exploitation de base minimal (Alpine, Ubuntu, etc.)
  • Les bibliothèques et dépendances
  • Le code source de l'application
  • Les outils et configurations spécifiques
📐 Structure en couches
Une image est constituée d'une série de couches en lecture seule. Chaque couche représente une instruction du Dockerfile et n'est qu'un ensemble de différences par rapport au calque précédent.

Image vs Conteneur

Une image = modèle statique en lecture seule. Un conteneur = instance active et exécutée de cette image.

  • Lorsqu'une image crée un conteneur, Docker ajoute une couche inscriptible au-dessus des couches de l'image
  • Les images peuvent exister sans conteneurs, mais un conteneur doit exécuter une image
  • On peut exécuter n'importe quel nombre de conteneurs à partir d'une image

Stratégie Copy-On-Write (COW)

Quand un conteneur modifie un fichier de l'image, il copie ce fichier dans sa propre couche privée et y apporte les modifications. L'image de base reste intacte.

🎯 Analogie Orientée Objet
Image ≈ Classe | Couches ≈ Héritage | Conteneur ≈ Instance

Namespaces des Images

Un namespace identifie de manière unique chaque image et l'organise dans une catégorie.

1. Images officielles (root-like)

Réservées à Docker Inc. Exemples : ubuntu, alpine, nginx, redis, postgresql

2. Images utilisateurs

Préfixe = nom d'utilisateur/organisation. Ex : jpetazzo/clock

3. Images auto-hébergées

Hébergées en dehors de Docker Hub. Incluent le nom d'hôte/IP et éventuellement le port du registre.

Création d'images Docker

Méthode 1 : FROM scratch

Image vide, ultra-légère. Idéale pour binaires autonomes.

FROM scratch
COPY hello /
CMD ["/hello"]
FROM scratch vs FROM ubuntu:18.04
  • FROM scratch — quelques Ko, une seule couche, aucun OS. Pour binaires statiques compilés.
  • FROM ubuntu:18.04 — ~30 Mo, multiples couches, outils OS complets. Pour applications nécessitant des packages système.

Méthode 2 : À partir d'un Dockerfile (build)

Méthode la plus utilisée — un script texte décrivant étape par étape la construction. 

docker build -t mon_image:v1 .
Context de build
Le . dans docker build -t mon_image . désigne le répertoire de contexte envoyé au daemon Docker : 
/mon-projet/
├── Dockerfile
├── app.py
├── requirements.txt
└── config/

Méthode 3 : À partir d'un conteneur (commit)

Capture l'état d'un conteneur modifié dans une nouvelle image.

docker commit [ID_conteneur] [nom_image]

Instructions du Dockerfile (DSL)

Instruction Rôle Exemple
FROM Image de base FROM ubuntu:20.04
LABEL Métadonnées LABEL version="0.1"
ENV Variables d'environnement ENV APP_ENV=production
USER Utilisateur par défaut USER appuser
WORKDIR Répertoire de travail WORKDIR /app
ARG Variables de construction ARG THEARG="foo"
COPY Copier fichiers locaux → image COPY ./source /app
ADD Comme COPY + extraction archives/URL ADD app.tar.gz /app
RUN Exécuter commandes à la construction RUN apt-get install -y python3
CMD Commande par défaut (remplaçable) CMD ["echo","Hello"]
ENTRYPOINT Commande principale (toujours exécutée) ENTRYPOINT ["echo"]
EXPOSE Ports utilisés EXPOSE 80
VOLUME Volumes persistants VOLUME /data
CMD vs ENTRYPOINT
CMD — commande par défaut, remplacée si on passe une commande au docker run.
ENTRYPOINT — commande principale, toujours exécutée. Les args de CMD ou du docker run sont ajoutés en paramètres.

Multi-Stage Build

Technique avancée utilisant plusieurs FROM dans un même Dockerfile pour créer des images légères.

Sans multi-stage (mauvaise pratique)

FROM node:14
WORKDIR /app
COPY . .
RUN npm install && npm run build
CMD ["npx", "http-server", "dist"]
# Image lourde avec Node.js + toutes les dépendances

Avec multi-stage (bonne pratique)

# Étape 1 : Build
FROM node:14 AS build-stage
WORKDIR /app
COPY . .
RUN npm install && npm run build

# Étape 2 : Image finale légère
FROM nginx:alpine
COPY --from=build-stage /app/dist /usr/share/nginx/html
# ~30 Mo au lieu de centaines de Mo
Principes clés
  • Multiples FROM pour segmenter les étapes
  • COPY --from= pour transférer les artefacts entre étapes
  • Image finale = uniquement les ressources nécessaires
  • Disponible depuis Docker 17.05+

Système de cache intelligent

Docker met en cache chaque étape du build sous forme de snapshot. Si aucun changement n'est détecté, il réutilise les couches en cache pour accélérer la construction.

⚠️ Attention au cache
Les commandes externes comme RUN apt-get update ne seront pas ré-exécutées même si les dépôts ont changé. Pour forcer un rebuild complet : 
docker build --no-cache -t mon_image .

Commandes essentielles pour les images

Commande Description
docker search <nom> Rechercher sur Docker Hub (affiche NAME, DESCRIPTION, STARS, OFFICIAL, AUTOMATED)
docker image pull <nom>:<tag> Télécharger une image (ex: docker image pull mysql:8.1). Vérifie le cache local d'abord.
docker image history <nom> Historique des couches : instructions (CMD, RUN, ADD), date de création, taille par couche
docker images Lister les images locales
docker rmi <nom>:<tag> Supprimer une image
docker build -t <tag> . Construire une image depuis un Dockerfile
docker commit <ID> <nom> Créer une image depuis un conteneur
Chapitre 4

Gestion des Volumes

Introduction : Persistance des données

Par défaut, les conteneurs sont éphémères — toutes les données sont perdues à la suppression. Pour la persistance, Docker propose des mécanismes de stockage.

⚠️ Problème
Si un conteneur MySQL est supprimé sans stockage persistant, toutes les données utilisateurs sont définitivement perdues.

Overlay2 Filesystem (UnionFS)

📦 Définition
Le overlay2 filesystem crée l'illusion de fusionner le contenu de plusieurs répertoires en un seul sans modifier les sources d'origine, en se basant sur le principe Copy-On-Write (COW).

Fonctionnement sous Docker

  • Modification — copie du fichier de lowerdir (lecture seule) vers upperdir (lecture-écriture)
  • Création — fichier placé uniquement dans upperdir
  • Suppression — un whiteout file est créé dans upperdir pour masquer le fichier (lowerdir reste intact)
  • Couches en lecture seule — protégées, jamais modifiées
  • Suppression du conteneur — upperdir disparaît → toutes les modifications sont perdues

Bind Mounts

📂 Définition
Mécanisme qui stocke les données d'un conteneur directement dans un répertoire de l'hôte. Le répertoire peut être situé n'importe où sur le système.

Avantages

  • Accessibilité directe des fichiers depuis l'hôte
  • Performances améliorées (accès direct)
  • Persistance après suppression du conteneur
  • Partage facile entre plusieurs conteneurs

Syntaxe

docker run -v /chemin/hote:/chemin/conteneur image

Exemple pratique

mkdir -p /home/bindmount/test
echo "hello" > /home/bindmount/test/fichier1.txt
docker run -it --name C1_bind -v /home/bindmount/test:/data alpine

Volumes de données

Gérés par Docker lui-même, stockés dans /var/lib/docker/volumes/.

Caractéristiques clés

  • Indépendance du conteneur — données préservées même si le conteneur est supprimé
  • Partage entre conteneurs — plusieurs conteneurs accèdent au même volume
  • Réduction de la taille — données gérées à l'extérieur du conteneur
Critère Volumes 🗄️ Bind Mounts 📂
Gestion Automatique par Docker Manuelle par l'utilisateur
Isolation Isolé du FS hôte Lié directement à l'hôte
Emplacement /var/lib/docker/volumes/ N'importe où sur l'hôte
Suppression docker volume rm Le dossier reste sur l'hôte

Volumes anonymes

Créés sans nom explicite. Docker génère un ID automatique.

docker run -it --name container_anon -v /data alpine

Cas d'utilisation : données temporaires, tests, cache.

# Voir le volume généré automatiquement
docker volume ls

# Inspecter les détails de montage
docker inspect container_anon

Volumes nommés

Volumes explicitement créés avec un nom spécifique. Meilleure gestion et partage. 

docker volume create mon_volume
docker run -it --name container_named -v mon_volume:/data alpine

Modes d'accès

  • rw (par défaut) — lecture et écriture
  • ro — lecture seule : -v mon_volume:/data:ro 
# Lecture-écriture (défaut)
docker run -d --name conteneur_rw -v mon_volume:/app/data alpine

# Lecture seule
docker run -d --name conteneur_ro -v mon_volume:/app/data:ro alpine
Critère Nommés Anonymes
Persistance Données persistent Perdues à la suppression du conteneur
Partage Idéal multi-conteneurs Un seul conteneur typiquement
Gestion Facile à sauvegarder/restaurer Difficile (pas de nom)
Flexibilité Compatible Compose/Swarm Tâches temporaires
Chapitre 5.1

Conteneurs & Réseaux

Pourquoi mettre les conteneurs en réseau ?

Les conteneurs sont isolés par défaut. Mais les applications modernes ont besoin que plusieurs services communiquent (web, BDD, microservices).

  • Un conteneur doit exposer un service (port/IP spécifique)
  • Accéder à des services externes (API, BDD, web)
  • La communication se fait via un réseau virtuel
  • Dans un réseau Docker, un conteneur a une adresse IP et optionnellement un hostname
⚠️ Adresse IP des conteneurs
Un conteneur reçoit une adresse IP uniquement quand il est en cours d'exécution. Lorsqu'il s'arrête, il perd son adresse IP. Au redémarrage, il peut recevoir une nouvelle IP, sauf si une IP statique lui a été attribuée.

Docker Network Drivers

Driver Description Usage
Bridge Réseau local de type pont. Driver par défaut. Conteneurs isolés du reste du réseau. Conteneurs standalone sur un même hôte
Host Supprime l'isolation réseau. Le conteneur utilise directement l'interface de l'hôte. Accès direct au réseau hôte (risques sécurité)
Overlay Réseau distribué sur plusieurs hôtes Docker. Routage transparent. Docker Swarm, multi-hôtes
None Aucune interface réseau. Conteneur totalement isolé. Sécurité maximale, aucune communication

Commandes réseau de base

Commande Description
docker network ls Lister les réseaux
docker network create --driver bridge mon_reseau Créer un réseau
docker network inspect <nom> Inspecter un réseau
docker network connect <réseau> <conteneur> Connecter un conteneur
docker network disconnect <réseau> <conteneur> Déconnecter un conteneur
docker network rm <nom> Supprimer un réseau
docker network prune Supprimer tous les réseaux non utilisés
Réseaux par défaut
Docker crée automatiquement 3 réseaux : bridge (bridge), host (host), none (null). Les réseaux none et host ne peuvent pas être créés manuellement.

Default Bridge Network

  • Créé par défaut, connecté à l'interface docker0
  • Chaque conteneur reçoit une IP dans le sous-réseau de docker0
  • Communication entre conteneurs uniquement par adresse IP (pas de résolution DNS)
  • Non accessible depuis l'extérieur sans mappage de port

Mappage de port

docker run -p <HOST_PORT>:<CONTAINER_PORT> <IMAGE>

Limitations du Default Bridge

  1. Pas de résolution DNS — communication uniquement par adresse IP
  2. Pas de sous-réseau personnalisé — docker0 est imposé
  3. Faible sécurité/isolation — tous les conteneurs se voient
  4. Pas de connect/disconnect dynamique — il faut arrêter et recréer le conteneur

User-Defined Bridge Network

Créé par l'utilisateur. Offre la résolution DNS automatique entre conteneurs. 

docker network create --driver=bridge --subnet=192.168.1.0/24 --gateway=192.168.1.1 mon_reseau
docker run --net=mon_reseau --name container1 alpine
💡 Résolution DNS
Avec un User-Defined Bridge, ping container1 fonctionne depuis un autre conteneur du même réseau — pas besoin de connaître l'IP !

Default vs User-Defined Bridge

Critère Default Bridge User-Defined Bridge
DNS Non (IP uniquement) Oui (résolution automatique)
Personnalisation Limitée Subnet, gateway, règles
Isolation Faible (tous les conteneurs se voient) Meilleure (réseau dédié)
Dynamique Pas d'ajout/retrait à la volée Connect/disconnect à la volée
Usage Tests simples Production, multi-conteneurs

Réseau Host

Le conteneur utilise directement l'interface réseau de l'hôte. Pas d'isolation réseau. 

docker run --net=host --name ng nginx

Accessible directement via localhost de la machine hôte.

Exemple : Déploiement WordPress avec réseau

# 1. Créer un réseau user-defined
docker network create --driver bridge Netw1

# 2. Lancer MySQL sur le réseau
docker run -d --name db --network Netw1 \
  -e MYSQL_ROOT_PASSWORD=example \
  -e MYSQL_DATABASE=mydb \
  -e MYSQL_USER=esprit1 \
  -e MYSQL_PASSWORD=esprit1 mysql:5.7

# 3. Lancer WordPress sur le même réseau
docker run -d --name myapp --network Netw1 -p 8081:80 \
  -e WORDPRESS_DB_HOST=db \
  -e WORDPRESS_DB_NAME=mydb \
  -e WORDPRESS_DB_USER=esprit1 \
  -e WORDPRESS_DB_PASSWORD=esprit1 wordpress:latest

# 4. Accéder : http://localhost:8081
💡 Grâce au User-Defined Bridge
WordPress utilise db (nom du conteneur) comme hôte BDD — la résolution DNS fonctionne automatiquement.
Chapitre 5.2

Docker Compose

Introduction à Docker Compose

🐙 Qu'est-ce que Docker Compose ?
Outil qui permet de définir et exécuter des applications multi-conteneurs avec un fichier YAML. Il orchestre plusieurs services, réseaux et volumes de manière déclarative.

Les 3 étapes de Docker Compose

  1. Définir l'environnement — Dockerfile pour chaque service
  2. Décrire les services — fichier docker-compose.yml
  3. Exécuter — docker compose up

Fonctionnalités

  • Construction et démarrage automatique des services

Chapitre 5.3

Docker Swarm

Introduction à Docker Swarm

🐝 Définition
Docker Swarm est la solution native d'orchestration de Docker. Il transforme un ensemble d'hôtes Docker en un seul hôte virtuel, permettant le clustering, la haute disponibilité et le scaling automatique.

Compose vs Swarm

  • Docker Compose — gestion locale, développement et tests
  • Docker Swarm — gestion distribuée multi-machines, production

Docker Swarm vs Kubernetes

Caractéristique Docker Swarm Kubernetes
Intégration Natif dans Docker Environnement séparé
Installation Facile, homogène Complexe, varie selon OS
Scaling Pas d'auto-scaling Auto-scaling, auto-réparation
Surveillance Pas intégrée (outils tiers) Intégrée
Load balancing Automatique via DNS Nécessite Nginx Ingress
Usage Besoins simples, débutants Projets complexes, entreprise

Architecture : Nœuds Manager & Worker

Nœud Manager

Responsable de la gestion et orchestration du cluster :

  • Gestion de l'état — maintient la cohérence et synchronisation
  • Planification des services — placement sur les Workers
  • API HTTP — interaction avec le cluster
  • Distribution des tâches aux Workers
  • Haute disponibilité — nombre impair recommandé

Composants du Manager

Composant Rôle
API Docker Accepte les commandes et crée les services
Orchestrator Compare état actuel vs souhaité, crée les tâches
Allocator Assigne les adresses IP aux tâches
Dispatcher Assigne les tâches aux Workers
Scheduler Décide quel Worker exécute quelle tâche
Raft Consensus — élit le Leader, synchronise l'état du cluster

Nœud Worker

  • Exécute les conteneurs assignés par le Manager
  • Ne participe pas à la gestion du cluster
  • Composants : Worker Agent (récupère les tâches) + Executor (lance les conteneurs)
⚠️ Règle importante
Un Swarm peut avoir un seul Manager, mais un Worker nécessite toujours un Manager.

Consensus Raft : Leader & Reachable

Lorsqu'il y a plusieurs Managers :

  • Leader — manager principal, responsable de l'orchestration et des décisions
  • Reachable — fait partie du quorum, peut participer mais ne dirige pas sauf si nécessaire
Élection automatique
Si le Leader tombe en panne, Raft élit automatiquement un nouveau Leader parmi les nœuds Reachable. Les Managers doivent être en nombre impair pour le consensus.

Services & Tâches

Services

Un service définit comment les conteneurs fonctionnent dans le cluster. Il spécifie :

  • L'image Docker, les ports exposés, le nombre de réplicas
  • Le réseau overlay, les limites CPU/mémoire
  • La politique de rolling update

Tâches (Tasks)

Une tâche = une instance de conteneur lancée par un service sur un Worker. États possibles :

  1. Assigned — attribuée à un nœud
  2. Preparing — en cours de préparation
  3. Running — en cours d'exécution
  4. Complete — terminée avec succès
  5. Failed — échec
  6. Replaced — remplacée pour maintenir l'état souhaité
Relation Service → Tâche → Conteneur
Service (3 réplicas) → 3 tâches créées → 3 conteneurs exécutés sur les Workers disponibles.

Gestion du Swarm

Initialiser un Swarm

docker swarm init --advertise-addr <IP_MANAGER>

Ajouter des nœuds

# Worker
docker swarm join --token SWMTKN-xxx IP_MANAGER:2377

# Récupérer le token worker
docker swarm join-token worker

# Récupérer le token manager
docker swarm join-token manager

Vérifier l'état & quitter

docker info                      # Infos du Swarm
docker swarm leave               # Quitter (worker)
docker swarm leave --force       # Désactiver complètement

Gestion des nœuds

Commande Description
docker node ls Lister tous les nœuds
docker node inspect <nom> Inspecter un nœud
docker node update --availability drain <nom> Mettre en maintenance
docker node update --availability active <nom> Remettre actif
docker node update --label-add key=value <nom> Ajouter un label
docker node promote <nom> Worker → Manager
docker node demote <nom> Manager → Worker
docker node rm <nom> Supprimer un nœud

Disponibilité des nœuds

  • active — peut exécuter des tâches
  • pause — ne reçoit plus de nouvelles tâches, garde les existantes
  • drain — vide de toutes ses tâches

Catégories de labels

Les labels permettent de catégoriser les nœuds pour optimiser l'orchestration :

  • role — frontend, backend, db
  • environment — dev, staging, prod
  • hardware — gpu, ssd
  • os — linux, windows
  • datacenter, region, backup, security_level

Docker Stack

📚 Définition
Un stack = ensemble de services déployés ensemble comme une application unique, définis dans un docker-compose.yml.

docker service vs docker stack

Critère docker service docker stack
Portée Un service à la fois Multi-services (fichier Compose)
Configuration Via la commande Via docker-compose.yml
Mise à jour docker service update docker stack deploy

Commandes docker stack

Commande Description
docker stack deploy -c docker-compose.yml <nom> Déployer un stack
docker stack ls Lister les stacks
docker stack services <nom> Services d'un stack
docker stack ps <nom> Conteneurs d'un stack
docker stack rm <nom> Supprimer un stack

Exemple : Stack Nginx + MySQL

version: "3.8"

services:
  web:
    image: nginx:latest
    deploy:
      replicas: 3
      resources:
        limits:
          cpus: "0.5"
          memory: "512M"
      restart_policy:
        condition: on-failure
    ports:
      - "80:80"

  db:
    image: mysql:5.7
    environment:
      MYSQL_ROOT_PASSWORD: example
    deploy:
      replicas: 1
      resources:
        limits:
          cpus: "0.5"
          memory: "512M"
      restart_policy:
        condition: on-failure
docker stack deploy -c docker-compose.yml appstack
docker stack services appstack
docker stack ps appstack
docker stack rm appstack
